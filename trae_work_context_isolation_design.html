DYShop · Trae Work Context Isolation

复杂一人公司工作流的上下文隔离方案

这份说明针对当前 Trae Work 对话框里“先加载 AGENTS.md，再逐级进入 workflow / agent / skill”的使用方式， 梳理当前上下文累计的风险、你真正想要的隔离式子工作流模型，以及在项目侧和平台侧分别需要做的改造。

一、现状判断

当前大概率是累计式上下文

Trae Work 对话更像一条连续线程

你在一个 Trae Work 对话框里发起任务后，后续所有分流、加载 workflow、加载 agent、执行开发和 QA， 通常都发生在同一个对话上下文里。除非平台或工具显式创建了新的子会话，否则已读过的规则、用户消息、 中间输出、文件片段和执行摘要都会持续占用上下文窗口。

风险

递归分流越多，上下文越混

当前一人公司模型会从 CEO 到一级 workflow，再到二级 workflow，再到具体角色。每一层都可能加载自己的 workflow yaml、agent 文档、skill 文档和运行证据。如果这些都堆在同一个上下文里，会带来 token 膨胀、 角色串味、旧规则残留、职责边界漂移和 QA/验收跳步等问题。

关键判断： `.trae` 里的 workflow / agent 文档可以规定“应该怎么交接”，但不能单靠文档让 Trae Work 自动丢弃历史、 自动新开子上下文或自动切换 system prompt。真正的上下文隔离需要运行时能力支持。

二、你想要的目标形态

你要的不是简单少输出几行，而是一个“父工作流负责调度，子工作流独立闭环，最后结构化回传”的执行模型。

父 Agent / 上级 Workflow

只保留任务目标、事实来源、约束和期望输出。

CEO 部长 handoff 发起者
→

隔离子运行 Sub-run

只加载自己的 system prompt、入口 agent、workflow yaml 和裁剪后的必要历史。

独立上下文 局部闭环 少量历史
→

结构化结果回传

返回结论、证据、状态字段、风险和下一跳，不把完整内部过程倒灌回父上下文。

summary artifacts pass flags

三、当前模型和目标模型的差异

维度 当前累计式对话 目标隔离式子工作流
上下文 同一会话持续追加，历史和提示词不断累积。 每个子工作流有独立上下文，只拿必要摘要和输入。
System Prompt AGENTS、policy、多个 agent 文档可能同时在上下文里共存。 子运行只加载自身入口角色和本 workflow 相关提示词。
父子关系 父角色和子角色在同一对话里连续发言，容易职责串味。 父角色只发 handoff，子角色独立完成后返回 result packet。
证据保留 大量中间内容堆在聊天里。 完整证据进入 artifact / state，父上下文只拿摘要。
成本 层级越深 token 越多。 每层只携带必要输入，父级上下文保持稳定。
风险 旧规则、旧结论、其他角色职责容易影响当前角色。 角色边界更干净，但需要运行时支持子会话和回传协议。

四、推荐解决方案

1. 定义 Handoff Packet

父级 workflow 不把完整聊天历史传给子级，只生成结构化 handoff 包。

  • 任务目标
  • 已确认事实
  • 边界约束
  • 期望输出字段
  • 必要文件 / 接口 / 证据引用

2. 启动 Isolated Sub-run

子工作流以新的运行单元启动，只加载自己的入口角色和必要上下文。

  • 自己的 system prompt
  • 自己的 workflow yaml
  • 自己的 agent 文档
  • 裁剪后的上游摘要
  • 必要 artifacts

3. 返回 Result Packet

子工作流结束后只把结构化结果回传父级，避免完整过程倒灌。

  • 完成状态
  • 关键结论
  • 证据索引
  • 风险和阻断
  • 下一跳建议

六、上下文裁剪策略

子工作流应保留

  • 上游 handoff packet。
  • 当前 workflow yaml。
  • 入口 agent 文档。
  • registry 中目标 agent / skill / tool 的路径。
  • 与本任务直接相关的文件、接口、截图、日志。
  • 最近一轮必要的用户原话。

子工作流应丢弃

  • 父级完整聊天历史。
  • 其他 workflow 的完整 agent 文档。
  • 与当前任务无关的历史调试过程。
  • 已经被结构化摘要替代的长文本。
  • 旧的错误结论和已失效规则片段。
  • 非当前角色所需的 skill/tool 细节。

七、两种落地路径

理想方案

平台支持真正子会话

Trae Work / runtime 提供类似 spawn_subrun、start_child_workflow 的能力。 父级把 handoff packet 传进去，平台新建隔离上下文，执行完成后把 result packet 返回父级。

优点：上下文干净、角色边界清楚、token 成本可控。

要求：平台必须支持子运行、artifact 存储、结果回调和状态合并。

过渡方案

项目侧模拟上下文隔离

如果 Trae Work 暂时不支持真正子会话，可以在 `.trae` 规则里强制每次 handoff 先生成短摘要， 子 workflow 只允许读取摘要、自己的 workflow 和入口 agent。虽然物理上下文仍在同一会话里， 但行为上尽量约束为“软隔离”。

优点：不依赖平台改造，马上能降低噪音。

缺点：旧上下文仍可能被模型看到，隔离不彻底。

八、建议的运行时状态模型

对象 用途 关键字段
workflow_run 记录一次一级 workflow 执行。 run_id、workflow_id、parent_run_id、status
workflow_subrun 记录子工作流隔离执行。 subrun_id、entry_agent、context_policy、result_packet
handoff_packet 父级给子级的最小输入。 task_goal、facts、constraints、required_outputs
artifact 保存完整证据，避免塞回聊天。 artifact_id、type、path、summary
result_packet 子级回传给父级的结构化结果。 completed、pass_flags、summary、next_node

九、对当前 `.trae` 体系的改造建议

  1. 新增上下文策略字段。 在 workflow yaml 中加入 context_policy，声明进入该 workflow 时保留哪些内容、丢弃哪些内容。
  2. 新增 handoff 输入输出契约。 每条跨 workflow edge 都要求输出 handoff_packet，目标 workflow 只消费 packet。
  3. 入口 agent 增加隔离规则。 入口角色只读取自己的 workflow、自己的 agent 文档和 packet，不主动继承父级全部上下文。
  4. 把完整证据移到 artifact。 长日志、截图、diff、QA 记录不直接回填父会话，只返回引用和摘要。
  5. 父级只消费 result packet。 父级不复盘子级所有细节，只根据结果字段决定继续、回流、review 或归档。

十、示例：自优化 handoff 开发流

self_optimization_workflow
  diagnose:
    context:
      read:
        - AGENTS.md summary
        - orchestration-policy
        - self-optimization-workflow.yaml
        - self-optimization-agent.md
      produce:
        - development_request
        - handoff_packet

development_workflow sub-run
  context_policy:
    isolation: hard
    keep:
      - handoff_packet
      - development-workflow.yaml
      - technology-minister-agent.md
      - directly referenced files
    drop:
      - parent chat history
      - unrelated workflow docs
      - old failed reasoning
  return:
    - result_packet
    - artifact_refs
    - state_delta

十一、结论

你的判断是对的：当前这种“复杂递归分流 + 多角色加载”的一人公司工作流，如果一直在同一个 Trae Work 对话上下文里累计加载，会越来越重，也会更容易发生角色串味和旧规则干扰。

最合理的目标是：父级 workflow 只负责调度和验收，子 workflow 用隔离上下文独立闭环，最后通过结构化 result packet 回传。项目侧可以先把 handoff packet、result packet、context_policy 固化进 `.trae` 规则；如果 Trae Work 平台支持真实子会话，再把这些契约接到运行时能力上。
