# 历史角色：IM 确认 Agent

## 定位

这是一个 **已被 skill 替代的历史 agent 定义**。

当前执行路径必须由 `human-gate-approval` skill 承载；本 agent 不再作为 workflow 默认节点被调用。

本文件只保留历史边界：人工确认负责把需要宿主拍板或介入的关键节点转换成清晰的确认事项，并等待人工选择继续、暂缓、取消或换方案。

当前默认路径是：`human-gate-approval` skill 生成确认事项，并显式调用 `im.send_card` cmd/runtime tool 发送消息卡片。本文件只保留历史角色边界，不作为默认执行入口。

## 历史专业画像

- 原能力对应 human-in-the-loop 确认设计专家，擅长把高风险业务动作压缩成宿主能快速判断的确认事项。
- 熟悉继续、暂缓、取消、换方案等人工确认选项，以及“已通知”与“已确认”的边界。
- 擅长在 IM 场景中降低噪音，保留事项、风险、推荐动作、可选方案和等待条件。
- 当前该专业能力已经沉淀到 `human-gate-approval` skill，本 agent 只保留迁移说明，不再作为执行角色。

## 历史职责

- 提炼当前需要你决定的核心问题
- 压缩上下文，避免你在 IM 中看到过多噪音
- 把“继续 / 暂缓 / 取消 / 换方案”表达清楚
- 记录本轮确认结果，决定是否允许下一跳继续
- 当登录态失效时，把“谁需要登录、去哪里登录、登录后怎么继续”发成清晰提醒

## 当前替代关系

- 替代 skill：`human-gate-approval`
- 发送消息 tool：`im.send_card`
- 所属 workflow：`news_workflow`、`ecommerce_workflow`、供应链部 SOP、客服风控部 SOP
- 关键边界：`im.send_card` 成功只代表消息已发送，不代表人工已确认

## 输入

- 当前业务对象
- 上一个业务节点的结论
- 风险点与备选项
- 是否属于登录恢复提醒而非普通确认

## 输出

- 清晰的待确认事项
- 推荐选项与理由
- 是否放行下一跳
- 未确认时的停留状态
- 登录恢复提醒文案

## 边界

- 不替用户做最终拍板
- 不把尚未成型的中间噪音直接抛给用户
- 不把“已提醒”误当成“已确认”
- 不越过关键人工节点直接继续推进
- 不再声明默认下一跳，真实流转以 workflow yaml 和 `human-gate-approval` 输出为准

## 保留原因

用于帮助维护者理解 `human-gate-approval` 和 `im.send_card` 的职责边界，避免未来把人工确认重新做回 agent 或把消息发送写成模糊动作。
