# 宿主验收与功能测评细则

这份细则是 `host-acceptance-agent` 的最终验收口径，同时也为 `technology-minister-agent`、`ui-ux-review-agent` 和 `regression-validation-agent` 提供统一的测评参考。

## 目标

- 确保测评基于真实运行中的 web，而不是静态猜测
- 确保所有页面都被覆盖，而不是只看首页或单个 demo 页面
- 确保评分、通过、回流都有统一规则，方便 `host-acceptance-agent` 做最终验收，也方便开发工作流内各节点使用同一口径回流

## 启动要求

- 默认启动命令：`sh run.sh`
- 默认检查项：
  - 前端：`http://localhost:5174`
  - 后端：`http://localhost:8000`
  - Bridge：按项目依赖检查健康状态
- 默认关闭命令：`sh stop.sh`
- 若启动失败、接口不可达、关键依赖未就绪，本轮只允许输出“前置条件不满足”，不允许输出“测评通过”。
- 不允许为了测评方便改用 `npm run dev`、`vite`、`uvicorn`、`docker compose up/down` 等旁路命令启动或关闭当前项目。

## 页面清单

当前项目至少要覆盖以下页面与入口：

| 页面 | 路由 | 最低检查要求 |
| --- | --- | --- |
| 主页 | `/` | 首屏信息、导航可用、能进入业务页 |
| 资讯聚合 | `/project/news` | 内容列表、摘要卡片、滚动、空态/加载态 |
| 电商总览 | `/project/ecommerce/overview` | 数据卡、概览模块、侧边栏切换 |
| 货盘页 | `/project/ecommerce/products` | 列表或结果展示、主要操作按钮、状态反馈 |
| 流程看板 | `/project/ecommerce/flow/default` | 流程节点、页面结构、返回路径 |
| 素材工坊 | `/project/ecommerce/creative-studio` | 主操作、结果展示区、加载/失败反馈 |
| 1688 选品 | `/project/ecommerce/sourcing` | 列表浏览、主要交互、滚动稳定性 |

补充规则：

- 若页面内还能进入二级详情、抽屉、模态框、悬浮层，这些都算当前页面巡检范围。
- 若存在菜单可见但路由跳转异常，也算页面级功能问题。
- 若某页面是重定向页，也要检查重定向是否符合预期。

## 效果测评量表

每页 7 个维度，每项 `0-5` 分，总分 `35` 分。

| 维度 | 评分关注点 | 不通过信号 |
| --- | --- | --- |
| 信息层级与状态可见性 | 重点是否一眼可见；当前状态、加载态、空态、异常态是否可识别 | 不知道页面在表达什么；状态消失；反馈缺失 |
| 文案语义与业务贴合度 | 是否说人话；是否贴近“结果展示台”定位 | 内部术语多、执行台语义残留、占位文案明显 |
| 一致性与设计规范 | 导航、按钮、图标、色彩、卡片规则是否一致 | 同类元素表现冲突，页面像不同产品拼起来 |
| 视觉聚焦与简洁度 | 主操作、主结果、主指标是否聚焦；页面是否干净 | 信息噪音大、重点淹没、视觉节奏混乱 |
| 视觉 style 体验 | 是否具备高完成度审美、统一二次元语言、Chiikawa 气质与治愈感；风格是否真正服务内容 | 只有零散贴纸、没有统一风格；过度堆砌；可爱但不美；二次元元素与页面结构割裂 |
| 可读性与基础无障碍 | 字号、对比度、控件可辨认性、键盘可达性、语义是否达标 | 看不清、点不准、焦点不可达、文本难懂 |
| 感知性能与视觉稳定性 | 首屏速度、交互响应、滚动流畅、布局抖动 | 首屏拖沓、点击后迟滞、滚动卡顿、布局乱跳 |

### 视觉 style 体验细分

这一项必须严格打分，不能因为“素材本身很可爱”就放宽标准。推荐按以下 4 个子点综合判断：

| 子点 | 通过表现 | 严格扣分信号 |
| --- | --- | --- |
| 美感完成度 | 配色、留白、圆角、阴影、层次、组件密度都成熟统一 | 颜色散、版面松紧失衡、像半成品拼贴 |
| 二次元语义一致性 | 角色、贴纸、图标、文案、页面节奏来自同一套世界观 | UI 很商务，素材很二次元，语气很系统，彼此割裂 |
| Chiikawa 气质贴合 | 柔和、圆润、轻脆弱、治愈、可亲近，有“萌但不廉价”的感觉 | 过度饱和、噪音感强、廉价卖萌、攻击性强或太俗艳 |
| 风格服务功能 | 风格增强识别度与记忆点，不压住内容和操作 | 风格喧宾夺主，主信息更难看清，交互更难理解 |

### 分数解释

- `5` 分：表现稳定，几乎不需要调整
- `4` 分：整体可接受，只有轻微瑕疵
- `3` 分：勉强可接受，但已有明显改进空间
- `2` 分：问题明显，已影响页面质量
- `1` 分：问题严重，接近不可接受
- `0` 分：页面失真、严重不可用或无法判断

### 效果通过规则

- 单页面：`总分 ≥ 30`，且无单项低于 `3`
- 单页面附加要求：`视觉 style 体验` 必须 `≥ 4`
- 全站：页面平均分 `≥ 30`，阻断问题 `0`，重要问题不超过 `2`
- 全站附加要求：首页、`Overview`、`Products`、`CreativeStudio` 的 `视觉 style 体验` 均需 `≥ 4`

## 功能测评量表

功能测评不追求“功能越多越好”，而是确认页面该做的事情能不能顺畅完成。

| 维度 | 必做动作 | 不通过信号 |
| --- | --- | --- |
| 可进入 | 菜单进入、按钮进入、直达路由进入 | 路由错误、白屏、跳不过去 |
| 可操作 | 点击主按钮、切换标签、打开弹层、关闭弹层 | 点击无反应、状态错乱、关闭失败 |
| 可反馈 | 观察 loading、成功、失败、空态反馈 | 没反馈、不知道是否生效 |
| 可滚动 | 鼠标滚轮滚动、长页下探、浮动按钮共存 | 卡死、错滚、遮罩锁死、内容抖动 |
| 可恢复 | 返回、重试、关闭、撤销 | 一旦出错就卡住，无法继续 |
| 主任务可完成 | 完成页面最核心的那一步 | 关键流程走不通，结果不可信 |

### 页面最小交互闭环

每页至少执行一次：

1. 进入页面
2. 浏览首屏
3. 执行主操作
4. 滚动到中后段
5. 触发次级交互或返回
6. 确认页面仍处于可继续操作状态

### 功能通过规则

- 单页面：主任务成功率 `100%`，阻断问题 `0`
- 全站：所有必测页面主任务都可完成，且没有明显功能问题

## 明显功能问题判定

出现以下任意一种，可直接判定本页功能不通过：

- 页面打不开、白屏、崩溃
- 主按钮无效或路径错误
- 菜单高亮和实际页面不一致
- 打开弹层后无法关闭或关闭后页面异常
- 滚动导致内容错位、卡死、长时间无响应
- 异步请求一直 loading，没有失败说明，也没有重试入口
- 页面展示结果和用户动作不一致，导致用户误判

## 问题分级

| 级别 | 含义 | 编排建议 |
| --- | --- | --- |
| 阻断 | 主任务无法完成或页面不可用 | 立即回流对应开发或 QA 节点，暂停放行 |
| 重要 | 主链路还能走，但质量明显不达标 | 进入前端/后端修复链，修后验收 |
| 一般 | 不影响主链路，但影响体验与完整度 | 记录到后续优化批次 |

## 回流模板

当页面不通过时，至少回传以下内容给 `host-acceptance-agent` 的验收摘要，并同步给 `technology-minister-agent` 重新分派：

- 页面名 / 路由
- 问题类型：效果 / 功能 / 混合
- 复现步骤
- 预期结果
- 实际结果
- 严重级别
- 建议下一跳
  - 偏视觉、结构、前端交互：`frontend-development-agent`
  - 偏数据、接口、鉴权、宿主依赖：`backend-development-agent`
  - 偏信息层级、导航认知、视觉策略：`ui-ux-review-agent`
  - 修复后统一复核：`regression-validation-agent`

## 外部依据

- 可用性启发式参考 Nielsen Norman Group 的 10 条可用性原则
- 视觉美感与“美即好用效应”参考 Nielsen Norman Group 对 `aesthetic-usability effect` 的说明：视觉吸引力会影响用户感知，但不能掩盖严重可用性问题
- 视觉 style 的结构判断参考 Nielsen Norman Group 的视觉设计 5 原则：`scale / hierarchy / balance / contrast / Gestalt`
- kawaii / 二次元 / Chiikawa 风格判断参考关于 `kawaii aesthetics` 的研究：其核心不是单一“可爱贴纸”，而是围绕 `cute / sweet / tender / innocent / soft / small / lovely` 等复合气质，以及圆润、柔和、易产生情感依附的视觉特征
- 基础无障碍参考 W3C WCAG 的 `Perceivable / Operable / Understandable / Robust`
- 感知性能参考 Google Core Web Vitals 的 `LCP / INP / CLS`

这些依据用于决定评分维度，不要求在本地开发阶段机械追求满分，但若页面明显违背这些基础原则，不得给出“通过”结论。
